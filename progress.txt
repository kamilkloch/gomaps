## Codebase Patterns

- **Server entry:** `server/src/index.ts` exports `app` (Express instance) for testability
- **Test framework:** Vitest in server/, tests in `server/tests/`
- **Client build:** `tsc -b && vite build` (project references)
- **Server build:** `tsc` (standard compilation)
- **Monorepo:** npm workspaces with `server` and `client` packages
- **Express version:** Express v5 (no need for @types/express body parsing)
- **DB layer:** `server/src/db/` â€” one file per entity, `schema.ts` for init, `types.ts` for interfaces, `index.ts` re-exports all
- **DB tests:** Use `createDatabase(tempPath)` for isolated test DBs; snake_case columns mapped to camelCase interfaces
- **Route files:** `server/src/routes/` â€” one file per resource, named export of Router (e.g., `projectsRouter`)
- **API integration tests:** Use `supertest` with `process.env.DB_PATH` set before importing app; cleanup with `closeDatabase()` + `unlinkSync`
- **Client routing:** React Router v7 with `BrowserRouter`; pages in `client/src/pages/`; named exports
- **Google Maps:** `@vis.gl/react-google-maps` with `<APIProvider apiKey={...}>` + `<Map>` pattern; key from `VITE_GOOGLE_MAPS_API_KEY`
- **Google Maps imperative overlays:** For `google.maps.Rectangle` and map listeners, keep imperative map objects in child components using `useMap()`, then pass updates to page state via callbacks
- **Map naming collision:** In React files importing the `Map` component, use `globalThis.Map` for JS map collections (`new globalThis.Map(...)`) to avoid TypeScript resolving to the component symbol
- **Explorer markers + clustering:** Manage `google.maps.Marker`, `google.maps.Circle`, and `MarkerClusterer` refs inside a dedicated `useMap()` controller; on data updates, diff markers by place id and refresh cluster membership (`clearMarkers()` + `addMarkers()`) to keep map state in sync with React state
- **Explorer virtualized selection sync:** Keep a single `selectedPlaceId` source of truth across map/table/detail state, and when selection changes, scroll the virtualized table viewport only if the selected row index is outside the current visible range
- **Explorer fuzzy search:** Drive global search from a debounced input state (~300ms) and run Fuse.js against a normalized per-place index (name/address/category/amenities/review text); lazily preload reviews only while a query is active so initial Explorer loads are not blocked by per-place review fanout.
- **Explorer filter-state URL sync:** Keep filter controls in local React state, parse initial values from `useSearchParams`, and mirror updates back into query params (only non-default values) so filters survive refresh/deep links without polluting routes.
- **Scraper modules:** `server/src/scraper/` â€” keep API clients isolated per provider (e.g., `places-api.ts`) and return parsed DB-ready payloads instead of raw API JSON
- **Tiling module structure:** keep `server/src/scraper/tiling.ts` split into pure bounds math (`generateTiles`, `subdivideTile`, `shouldSubdivide`) and thin DB-backed progress helpers (`initializeTilesForRun`, `markTileCompleted`, `subdivideTileInRun`, `getRunTileProgress`)
- **Scrape engine + counters:** when orchestrating runs in `server/src/scraper/engine.ts`, create/subdivide/complete tiles via tiling helpers instead of raw tile CRUD so `scrape_runs.tiles_*` counters stay in sync
- **Scrape route orchestration:** keep background scrape state (`pauseRequested`, in-flight task handles, SSE subscribers) module-scoped in `server/src/routes/scrape.ts`, and expose lightweight test hooks (`set...ForTests`, `reset...ForTests`) to make async integration tests deterministic
- **Places API requests:** Use `X-Goog-Api-Key` + `X-Goog-FieldMask` headers and wrap network calls in `Effect.tryPromise` with typed `ScrapeError`
- **Website classification:** Normalize website hostnames with URL parsing (add `https://` fallback for protocol-less inputs), then classify via suffix matching so `www.booking.com` and other subdomains map to the base domain allowlist
- **Place IDs:** Use Google Place IDs from Places API (`place.id` or resource name `places/{id}`), and persist canonical map links in `google_maps_uri`
- **Effect-TS:** All DB functions return `Effect<A, DbError | NotFoundError>`. Routes use `Effect.runPromise` at boundary. Scraper uses `Effect.runSync` for DB calls since better-sqlite3 is synchronous. Typed errors in `server/src/errors.ts`: `DbError`, `NotFoundError`, `ValidationError`, `ScrapeError`. Schema validation via `Schema.decodeUnknown` from `effect` package.
- **Effect Schema array mutability:** Values decoded through `Schema.decodeUnknown` can infer readonly arrays; clone (`[...value]`) before passing arrays into mutable DB input types (`CreatePlaceInput`) to satisfy strict TypeScript checks.
- **Playwright harness layout:** Keep reusable e2e primitives in `client/e2e/{fixtures,utils,pages,tests}`; centralize strict locator fallback (`data-testid` â†’ role â†’ text), network/visibility wait helpers, screenshot helper, and console/page-error fail-fast logic inside shared fixtures/utilities.
- **Playwright API stubbing:** For deterministic UI flows, install a single stateful `page.route('**/api/**', ...)` handler per test and mutate in-memory project/run/place fixtures so create/delete/start/pause/search behaviors can be verified without depending on external APIs.
- **Playwright route matching:** For nested endpoint mocks (e.g., `/api/scrape/:runId/...`), use glob patterns like `**/api/scrape**` (not `**/api/scrape*`) and read methods via `route.request().method()` to avoid silently missing subpath requests.
- **Playwright fixture seeding:** `POST /api/test/seed-fixtures` accepts optional `existingProjectId` to append runs/tiles/places onto an existing seeded project; use this to create mixed run histories for cross-view regressions.
- **Playwright env loading:** Parse root `.env` in `client/playwright.config.ts` before web server startup so both `VITE_GOOGLE_MAPS_API_KEY` and `GOOGLE_MAPS_API_KEY` flow into client/server e2e processes consistently.
- **Playwright waits on SSE pages:** Avoid `networkidle` waits on Setup-like pages with long-lived SSE requests; wait for deterministic UI locators/state transitions instead.
- **Playwright pending-request UI assertions:** For loading-state copy tied to slow API responses (e.g., "Saving boundsâ€¦"), block the matching `page.route` fulfillment with a deferred promise, assert the temporary UI state, then resolve the request.
- **Playwright map debug snapshots:** For map overlays/markers, expose Playwright-only hidden JSON snapshots (gated by `VITE_E2E_TEST_MODE`) and assert those snapshots instead of brittle `.gm-style` internals.
- **Explorer map interaction E2E hooks:** For marker-click/map-click/zoom tests, expose a Playwright-only map controller (`window.__gomapsExplorerDebug`) plus hidden snapshots (`explorer-marker-debug`, `explorer-cluster-debug`, `explorer-selection-circle-debug`) instead of probing Google Maps DOM internals.
- **SQLite timestamp ordering:** When listing rows by timestamp fields (`created_at`, etc.), add a deterministic tie-breaker (`rowid DESC`) so rapid same-second inserts keep stable ordering in UI tests and cards.
- **Setup deep-link not-found handling:** For `/projects/:projectId/setup`, treat API `404` as a deterministic "Project not found." page state instead of generic load-error copy.
- **Explorer keyboard row selection:** Make table rows focusable (`tabIndex={0}`) with Enter/Space selection handlers, and keep key map/table containers discoverable via named region labels for accessibility-first E2E locators.
- **Route-level a11y landmarks:** Keep page roots as `<main>` landmarks and expose named regions (`role="region"` + `aria-label`) for Projects/Setup/Explorer panels so navigation + accessibility e2e tests can assert routing and keyboard behavior without brittle selectors.
